# Server 인증 API 구현

**날짜**: 2026-03-29
**스프린트**: Sprint 01
**상태**: ✅ 완료

---

## 무엇을 했는가

회원가입, 로그인, Access Token 재발급, 로그아웃 4개의 API 엔드포인트를 구현했다. 요청 검증(Zod), 비즈니스 로직(authService), DB 접근(userRepository), 토큰 처리(jwt 유틸)를 각 레이어로 분리해 역할을 명확히 나눴다. TypeScript 컴파일 에러 없이 서버가 정상 실행된다.

---

## 왜 이렇게 했는가

- **레이어 분리**: 라우터 → 컨트롤러 → 서비스 → 레포지토리 흐름을 엄격하게 지켰다. 서비스가 DB를 직접 호출하지 않고 레포지토리를 통하게 해서 나중에 DB를 교체하거나 테스트할 때 서비스 코드를 건드리지 않아도 된다.
- **AppError 클래스**: 비즈니스 에러(409 중복, 401 인증 실패 등)와 예상치 못한 서버 에러를 `errorMiddleware` 한 곳에서 구분해 처리한다. 컨트롤러가 에러를 직접 처리하지 않아도 된다.
- **로그인 실패 메시지 통일**: 이메일 없음과 비밀번호 불일치를 같은 메시지로 응답한다. 다르면 이메일 존재 여부가 노출되어 열거 공격(enumeration attack)에 취약해진다.
- **Refresh Token DB 검증**: 서명 검증만으로는 로그아웃된 사용자를 막을 수 없다. DB에 저장된 값과 일치해야만 재발급하도록 두 단계 검증을 거친다.
- **인덱스 서버 시작 시 생성**: `ensureIndexes()`를 bootstrap에서 호출한다. 이미 인덱스가 있으면 MongoDB가 무시하므로 매번 호출해도 안전하다.

## 작업 흐름 (Workflow)

1. bcrypt, jsonwebtoken 패키지 설치
2. `shared/types.ts`에 User 타입 추가
3. Zod 검증 스키마(`authValidator.ts`) 작성
4. DB 접근 레이어(`userRepository.ts`) 작성 — email 유니크 인덱스 포함
5. JWT 유틸(`jwt.ts`) 작성 — Access/Refresh 발급 및 검증
6. `AppError` 클래스 작성
7. `authService.ts` 작성 — register, login, refresh, logout
8. `authMiddleware.ts` 작성 — Bearer 토큰 검증
9. `authController.ts` 작성 — 요청/응답 처리
10. `routes/auth.ts` 작성 — 4개 엔드포인트 연결
11. `errorMiddleware.ts` 작성 — 통합 에러 핸들러
12. `index.ts` 수정 — 라우터 등록 + 인덱스 설정 + 에러 미들웨어 등록
13. TypeScript 컴파일 확인 → 에러 없음
