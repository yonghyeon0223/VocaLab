---

## 무엇을 했는가

users와 pendingVerifications 두 컬렉션의 DB 접근 레이어를 구현했다. 각 레포지토리가 필요한 인덱스를 직접 정의하고, 서버 시작 시 index.ts에서 호출해 준비 상태를 보장한다.

---

## 왜 이렇게 했는가

- **Refresh Token 해싱값 저장**: `insertUser`와 `updateRefreshToken`은 이미 해싱된 값을 받는다. 레포지토리는 해싱 책임을 지지 않는다. 해싱은 service에서 처리하고 레포지토리는 저장만 담당한다.
- **pendingVerifications upsert**: 같은 이메일로 재발송 요청이 오면 기존 document를 덮어쓴다. insert + update 두 번 치는 것보다 atomic하게 처리된다.
- **TTL 인덱스 24시간**: 인증하지 않고 방치된 pendingVerifications를 MongoDB가 자동으로 정리한다. 인증 완료 시에는 `deleteByEmail`로 즉시 삭제해 불필요한 데이터가 남지 않게 한다.
- **incrementLoginAttempts와 lockUser 분리**: 실패 카운트 증가와 잠금 설정을 별도 함수로 분리했다. service에서 "20회 도달 시에만 lock" 판단을 명시적으로 처리할 수 있다.

## 작업 흐름 (Workflow)

1. userRepository 작성 — findByEmail, insertUser, updateRefreshToken, incrementLoginAttempts, lockUser, resetLoginAttempts, ensureIndexes

3. index.ts에 두 레포지토리 인덱스 초기화 호출 추가
4. TypeScript 컴파일 확인 → 에러 없음
