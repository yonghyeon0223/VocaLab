## 무엇을 했는가

인증 관련 비즈니스 로직 전체(register, sendVerification, verifyEmail, login, refresh, logout)와 Zod 검증 스키마를 구현했다. TypeScript 컴파일 에러 없이 완성됐다.

---

## 왜 이렇게 했는가

- **`issueAndSendCode` 내부 함수 분리**: register와 sendVerification이 "코드 생성 → 해싱 → upsert → 발송" 흐름을 동일하게 쓴다. 중복 없이 한 곳에서 관리한다.
- **verifyEmail에서 userId 확보 후 토큰 발급**: insertUser 후 실제 userId로 토큰을 발급한다. 임시 userId로 토큰을 만들었다가 교체하는 이중 생성을 피했다. insertUser 시 refreshToken 필드는 빈 문자열로 넣고 바로 업데이트한다.
- **Refresh Token 해싱 저장**: 평문을 DB에 저장하면 DB 탈취 시 모든 세션이 위험해진다. bcrypt 해싱 후 저장해서 원본 복원을 막는다.
- **Rotation 전략**: refresh 성공 시 새 Refresh Token을 발급하고 기존을 즉시 폐기한다. 이전 토큰이 재사용되면 DB 값과 불일치 → 탈취로 간주 → 전체 폐기로 이어진다.
- **sendVerification 스팸 방지**: pendingVerifications에 없는 이메일은 200을 반환한다. 존재 여부를 알려주면 계정 열거 공격에 노출된다.
- **login 잠금 판단 순서**: `incrementLoginAttempts` 후에 newAttempts를 계산해 20회 여부를 판단한다. DB를 한 번 더 읽지 않고 메모리의 이전 값 +1로 계산한다.

---

## 작업 흐름 (Workflow)

1. authValidator.ts 작성 (register, sendVerification, verifyEmail, login, refresh 스키마)
2. authService.ts 초안 작성 중 refresh 함수에서 `findByEmail('')` 버그 발견
3. userRepository에 `findById` 추가 후 authService 재작성
4. TypeScript 컴파일 확인 → 에러 없음
